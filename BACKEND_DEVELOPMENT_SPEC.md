# 后端开发规范

后续所有后端开发默认遵守本规范。

## 1. 分层约束

- Controller 只负责收参与出参，不直接写业务逻辑
- Service 负责业务编排、校验、状态流转
- Repository 只负责数据访问
- DTO 负责接口输入输出
- Entity 不直接作为普通接口返回对象

## 2. 认证与权限

### 普通用户体系

- 用户表：`app_user`
- 会话表：`user_session`
- 登录接口：`/api/users/...`
- 只允许访问普通用户业务接口

### 管理员体系

- 管理员表：`admin_user`
- 会话表：`admin_session`
- 登录接口：`/api/admin/auth/...`
- 管理接口：`/api/admin/...`

### 基本原则

- 普通用户和管理员账号不能混用
- 普通用户不能升权为管理员
- 新管理员账号必须由已登录管理员创建
- 管理后台不能嵌入普通用户页面流程

## 3. 接口规范

- 统一返回 `ApiResponse<T>`
- 成功返回 `success=true`
- 失败返回明确 `code`、`message`
- Controller 参数使用 DTO，不直接暴露 Entity
- 需要登录态的接口从鉴权上下文取当前用户，不信任前端传入身份字段

## 4. 参数与返回

### 入参

- 新增、编辑、登录、审核等操作使用明确请求 DTO
- 不允许用散乱 `Map` 承接复杂请求
- 字段校验尽量集中在 Service

### 出参

- 普通业务返回 DTO
- 管理员接口返回管理员 DTO 或管理视图 DTO
- 敏感字段如密码哈希、盐值、会话哈希不能出现在返回体中

## 5. 错误码

- 使用 `ApiErrorCode`
- 新增错误场景时同步补充枚举与消息语义
- 错误信息必须可读，避免“系统异常”式空泛描述

## 6. 数据库与迁移

- 表结构变化必须通过迁移脚本维护
- 开发期允许保守兼容旧库，但新增结构必须同步写入 `db/migration`
- 表注释、关键字段注释需要补齐中文说明

## 7. 注释规范

- 类注释：说明职责边界
- 关键字段注释：说明真实业务含义
- 复杂业务分支可加简短注释解释原因
- 避免无意义注释

## 8. 命名规范

- Controller：`XXXController`
- Service：`XXXService`
- Repository：`XXXRepository`
- Request DTO：`XXXRequest`
- Response DTO：`XXXResponse`

## 9. 测试要求

- 修改认证、权限、审核、登录等主流程时，必须补测试
- 至少覆盖：
  - 登录成功/失败
  - 越权访问
  - 关键状态流转
  - 管理接口边界

## 10. 当前架构决策

- 管理员后台走独立页面入口 `admin.html`
- 普通用户前台走 `index.html`
- 管理员与普通用户会话完全独立
- 普通用户列表不应再展示管理员账号
