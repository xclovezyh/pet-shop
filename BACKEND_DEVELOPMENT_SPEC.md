# 后端开发规范

后续所有后端开发默认遵守本规范。

## 1. 分层约束

- Controller 只负责收参与出参，不直接编写业务逻辑
- Service 负责业务编排、校验、状态流转
- Repository 只负责数据访问
- DTO 负责接口输入输出
- Entity 不直接作为普通接口返回对象

## 2. 认证与权限

### 普通用户体系

- 用户表：`app_user`
- 会话表：`user_session`
- 登录接口：`/api/users/...`
- 只允许访问普通用户业务接口

### 管理员体系

- 管理员表：`admin_user`
- 会话表：`admin_session`
- 登录接口：`/api/admin/auth/...`
- 管理接口：`/api/admin/...`

### 基本原则

- 普通用户和管理员账号不能混用
- 普通用户不能升权为管理员
- 新管理员账号必须由已登录管理员创建
- 管理后台不能嵌入普通用户页面流程

## 3. 接口规范

- 统一返回 `ApiResponse<T>`
- 成功返回 `success=true`
- 失败返回明确的 `code`、`message`
- Controller 参数使用 DTO，不直接暴露 Entity
- 需要登录态的接口，从鉴权上下文获取当前用户，不信任前端传入身份字段

## 4. 入参与出参规范

### 入参

- 新增、编辑、登录、审核等操作使用明确的 Request DTO
- 不使用散乱 `Map` 承接复杂请求
- 参数校验优先集中在 Service，通用基础校验可放在 DTO 注解层

### 出参

- 普通业务统一返回 Response DTO
- 管理接口返回管理视角 Response DTO
- 密码哈希、盐值、会话哈希等敏感字段禁止出现在返回体中

## 5. 分页规范

- 列表接口默认支持 `page`、`size`
- `page` 从 1 开始
- `size` 必须有上限，避免一次返回过多数据
- 分页结果统一使用 `PageResponse<T>`

示例：

```json
{
  "items": [],
  "total": 0,
  "page": 1,
  "size": 10,
  "totalPages": 0,
  "hasNext": false,
  "hasPrevious": false
}
```

- 管理台列表接口优先分页
- 数据量继续增长时，应优先迁移为数据库层分页查询

补充：

- 树形基础资料可不走分页，但必须按层级结构返回，避免前端一次性平铺全部节点
- 地区库这类标准基础数据，优先只读展示与筛选，不优先开放后台人工维护

## 6. 错误码规范

- 使用 `ApiErrorCode`
- 新增错误场景时同步补充枚举与语义说明
- 错误信息必须可读，避免空泛描述

## 7. 数据库与迁移

- 表结构变更必须通过迁移脚本维护
- 开发期允许兼容旧库，但新增结构必须同步写入 `db/migration`
- 表注释、关键字段注释需要补齐中文说明
- 新增表时同步补充核心字段中文注释
- 引入国家或行业公开标准基础数据时，需在迁移或脚本中注明来源与版本

## 8. 注释规范

- 类注释说明职责边界
- 关键字段注释说明真实业务含义
- 复杂分支可加简短注释解释原因
- 避免无意义注释

## 9. 命名规范

- Controller：`XXXController`
- Service：`XXXService`
- Repository：`XXXRepository`
- Request DTO：`XXXRequest`
- Response DTO：`XXXResponse`

## 10. MVC 实践要求

- Controller 不直接操作 Repository
- Service 不直接暴露 Entity 给前端
- View 层需要的数据结构由 DTO 负责整形
- 公共转换逻辑可沉淀到专用 mapper 或装配方法中

## 11. 测试要求

- 修改认证、权限、审核、登录等主流程时必须补测试
- 至少覆盖：
  - 登录成功与失败
  - 越权访问
  - 关键状态流转
  - 管理接口边界
  - 分页参数与分页结果

## 12. 当前架构决策

- 管理后台使用独立入口 `admin.html`
- 普通用户前台使用 `index.html`
- 管理员与普通用户会话完全独立
- 普通用户列表中不再混入管理员账号概念
